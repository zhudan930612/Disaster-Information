# 灾害信息员大赛平台 · 应用架构与模块拆分

## 1. 文档定位

本文档承接《03-总体产品架构与核心业务流程》，将业务能力域映射为系统模块，明确模块边界、职责分工与协作关系，作为后续终端级功能设计、接口设计和开发分工的依据。

本文档重点回答：

- 系统由哪些模块组成
- 每个模块负责什么、不负责什么
- 模块之间如何协作以支撑核心业务流程
- 每个模块对应哪份后续专项设计文档

本文档不展开页面交互细节和接口字段清单，后续在终端级功能设计和接口设计中展开。

**约束文档参考：**
- 《灾害信息员竞赛平台产品上下文》
- 《灾害信息员培训与竞赛SaaS平台产品规划》
- 《03-总体产品架构与核心业务流程》

---

## 2. 设计目标

- 将业务架构中的能力域落地为可独立设计、可独立开发的系统模块
- 明确模块边界，避免功能遗漏或重复实现
- 建立模块与核心业务流程的映射关系，确保每条流程都有模块承载
- 建立模块与后续设计文档的映射关系，指导阅读和分工
- 基于国家培训竞赛平台现有能力进行改造，优先复用已有模块

## 3. 系统边界定义

### 3.1 系统范围内

| 模块类别 | 模块 | 说明 |
|---------|------|------|
| 用户触达层 | 学员竞赛小程序 | 面向灾害信息员学员的练习、参赛、查看入口 |
| 用户触达层 | 竞赛管理后台 | 面向省级/地市县/国家级管理员的赛事管理与查看后台 |
| 用户触达层 | 平台运营配置后台 | 面向我方运营/平台超级管理员的租户开通与配置后台 |
| 能力层 | 用户中心 | 身份映射、登录态、权限、组织人员管理、机构树管理（省-市-县-乡） |
| 能力层 | 赛事引擎 | 练习赛、正式赛事生命周期、阶段流转、晋级结算、答题过程（规则固定现有模板） |
| 能力层 | 题库引擎 | 题库分层管理（平台共用/省级自主/权限公开）、组题策略、题目版本、快照锁定 |
| 能力层 | 统计中心 | 成绩、排名、晋级、看板、报表、数据导出 |
| 能力层 | 协同网关 | 国家级平台对接、数据同步、次数发放触发、结果回流 |
| 能力层 | 配置中心 | 租户开通、组织配置、规则参数、学习联动规则、题库装载 |

### 3.2 系统范围外

| 系统 | 说明 |
|------|------|
| 国家级培训竞赛平台 | 外部协同平台，负责课程学习、视频播放、学习任务完成判定 |
| 全国灾害信息员数据库 / 报灾系统人员库 | 可选校验源，非本平台建设范围 |
| 短信/消息通知服务 | 外部通道服务，本平台只负责调用 |

### 3.3 复用边界

- 基于现有国家培训竞赛平台进行改造升级，不是从零新建
- 题库管理、竞赛管理、报名入口、竞赛结果、数据统计等现有能力优先复用
- 系统管理（角色、账号、导航）在现有基础上扩展机构管理
- 学习联动机制简化改造：从复杂任务完成判定简化为看视频得次数

## 5. 模块职责定义

### 5.1 用户中心（UC）

| 维度 | 内容 |
|------|------|
| **负责** | 国家级平台用户身份映射、登录态生命周期管理、角色权限判定、组织架构与人员数据维护（省-市-县-乡四级机构树）、用户基础信息缓存与更新、行政层级权限下沉 |
| **不负责** | 不管理赛事规则与赛事状态（赛事引擎管）、不直接对接国家级平台接口（协同网关管）、不生成排名报表（统计中心管） |
| **对外接口** | 用户登录/登出、身份映射查询、权限校验、组织架构查询、人员信息查询与更新、机构树管理 |

### 5.2 赛事引擎（MC）

| 维度 | 内容 |
|------|------|
| **负责** | 练习赛发起与次数扣减、正式赛事创建与发布、赛事阶段流转（初赛→复赛→决赛）、晋级结算与结果生成、答题过程管理、同卷锁定机制；赛事规则固定现有模板，不做自定义赛制编排 |
| **不负责** | 不管理用户身份（用户中心管）、不组题抽题（题库引擎管）、不生成统计看板（统计中心管）、不直接对接国家级平台（协同网关管） |
| **对外接口** | 创建赛事、发布赛事、查询赛事状态、发起练习赛、提交答题、查询晋级结果、阶段推进 |

### 5.3 题库引擎（QC）

| 维度 | 内容 |
|------|------|
| **负责** | 题库分层管理（平台共用题库/省级自主题库/题目权限公开）、组题策略执行、题目版本管理、场次题目快照生成与锁定、题型与知识点标签管理 |
| **不负责** | 不管理赛事生命周期（赛事引擎管）、不管理租户配置（配置中心管）、不直接面向用户展示（终端层负责） |
| **对外接口** | 组题请求、题库范围查询、题目详情查询（按权限）、快照查询、题库装载与绑定、题目权限配置 |

### 5.4 统计中心（SC）

| 维度 | 内容 |
|------|------|
| **负责** | 练习数据统计、赛事成绩汇总、排名计算、晋级结果生成、统计看板数据组装、报表导出、省级/地市县汇总数据生成 |
| **不负责** | 不管理原始答题过程（赛事引擎管）、不管理用户身份（用户中心管）、不直接对接国家级平台（协同网关管） |
| **对外接口** | 练习统计查询、赛事统计查询、排名查询、晋级结果查询、报表导出请求、省级汇总数据查询 |

### 5.5 协同网关（SG）

| 维度 | 内容 |
|------|------|
| **负责** | 国家级平台接口对接、用户基础数据同步、视频学习完成状态接收、额外答题次数发放触发、完整比赛结果回流、同步异常补偿与重试、协同日志记录 |
| **不负责** | 不管理用户登录态（用户中心管）、不管理赛事规则（赛事引擎管）、不直接展示给用户（终端层负责） |
| **对外接口** | 同步任务执行、学习完成状态接收、次数发放请求、比赛结果回流、同步重试、协同日志查询 |

### 5.6 配置中心（CC）

| 维度 | 内容 |
|------|------|
| **负责** | 租户开通与生命周期管理、组织架构配置（省-市-县-乡）、赛事能力开关、练习赛模板参数、正式赛事规则参数边界、答题次数基础规则、学习联动规则配置（看视频得次数）、题库装载与范围绑定 |
| **不负责** | 不直接执行业务逻辑（各能力模块负责）、不生成统计数据（统计中心管）、不直接对接国家级平台（协同网关管） |
| **对外接口** | 租户CRUD、组织配置、规则参数配置、模板参数配置、题库装载、学习联动规则配置 |

## 6. 模块间协作关系

### 6.1 练习赛与学习激励联动闭环

```
用户打开小程序
    │
    ▼
[用户中心] 登录态校验 + 身份映射
    │
    ▼
[赛事引擎] 发起练习赛请求
    │
    ├──► [题库引擎] 组题请求（平台共用题库）→ 返回题目集合
    │
    ├──► [用户中心] 查询剩余次数
    │
    ▼
次数充足?
    ├── 是 → [赛事引擎] 扣减次数 → 答题过程 → 提交答案
    │           │
    │           ▼
    │       [赛事引擎] 计算成绩 → [统计中心] 沉淀练习记录
    │
    └── 否 → [赛事引擎] 返回次数不足
                │
                ▼
            [协同网关] 查询学习任务配置
                │
                ▼
            用户跳转国家级平台观看视频
                │
                ▼
            [协同网关] ← 接收学习完成状态
                │
                ▼
            [配置中心] 匹配奖励规则
                │
                ▼
            [协同网关] → [用户中心] 发放奖励次数
                │
                ▼
            用户返回小程序继续练习
```

### 6.2 正式赛事闭环

```
[竞赛管理后台]
    │
    ▼
[赛事引擎] 创建赛事请求
    │
    ├──► [用户中心] 校验权限与组织范围
    │
    ├──► [配置中心] 读取赛事规则参数边界
    │
    ├──► [题库引擎] 确认题库范围可用（省级自主题库）
    │
    ▼
[赛事引擎] 赛事发布
    │
    ├──► [题库引擎] 生成并锁定题目快照（同卷）
    │
    ├──► [用户中心] 通知可参赛用户
    │
    ▼
用户参赛
    │
    ▼
[赛事引擎] 答题过程管理 → 阶段结束
    │
    ▼
[赛事引擎] 晋级结算请求
    │
    ├──► [统计中心] 成绩汇总与排名计算
    │
    ├──► [赛事引擎] 生成晋级结果
    │
    ▼
[赛事引擎] 发布阶段结果
    │
    ├──► [统计中心] 沉淀赛事结果
    │
    └──► [协同网关] 完整比赛结果回流国家级平台
```

### 6.3 SaaS 交付闭环

```
[平台运营配置后台]
    │
    ▼
[配置中心] 创建租户
    │
    ├──► [配置中心] 初始化组织架构与角色权限（省-市-县-乡）
    │
    ├──► [用户中心] 同步/导入人员基础数据
    │
    ├──► [题库引擎] 装载平台共用题库 + 省级自主题库
    │
    ├──► [配置中心] 配置练习赛模板参数
    │
    ├──► [配置中心] 配置正式赛事规则参数边界
    │
    ├──► [配置中心] 配置答题次数规则
    │
    └──► [配置中心] 配置学习联动规则（看视频得次数）
                │
                ▼
            [协同网关] 建立协同策略上下文
                │
                ▼
            [配置中心] 租户状态变更为"联调中"
                │
                ▼
            联调验证通过 → [配置中心] 租户状态变更为"已上线"
```

### 6.4 数据回流闭环

```
[赛事引擎] 练习结果 + 比赛结果
    │
    ▼
[统计中心] 统一数据沉淀
    │
    ├──► 个人竞赛记录（关联用户中心身份）
    │
    ├──► 赛事统计看板
    │
    ├──► 管理后台数据查询
    │
    └──► [协同网关] 完整比赛结果回流国家级平台
                │
                ▼
            [国家级平台] 接收结果数据
```

## 9. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 为什么基于国家平台改造而不是新建？ | 复用现有竞赛能力，最小成本交付 | 国家平台已有题库、竞赛管理、报名、结果、统计等成熟能力，改造比新建周期短、风险低 |
| 为什么把"协同"拆成独立网关模块？ | 不放进用户中心或赛事引擎 | 国家级平台对接涉及认证、协议转换、补偿重试，独立网关便于隔离外部变化、统一兜底处理 |
| 为什么统计中心独立？ | 不直接让赛事引擎承担统计 | 赛事引擎聚焦实时比赛过程，统计需要跨赛事聚合、离线计算和复杂报表，独立后互不影响 |
| 为什么配置中心合并了租户+规则？ | 不按租户和规则拆成两个模块 | 租户开通、组织配置、规则参数、学习联动本质都是"运营配置"，由同一批运营人员操作，合并降低交付复杂度 |
| 为什么用户中心独立于配置中心？ | 不把人员管理放进配置中心 | 用户中心聚焦身份、权限、登录态，是运行时高频依赖；配置中心聚焦初始化配置，是低频写操作。分离后用户中心可独立优化性能 |
| 为什么赛事引擎独立于题库引擎？ | 不合并为一个模块 | 赛事规则（时间、阶段、晋级）与题目供给（组题、版本、快照）是两个不同维度的变化点，分离后各自演进 |
| 为什么题库要分层管理？ | 不按单一题库管理 | 练习赛用平台共用题库保证一致性，正式赛事允许各省自主配置满足差异化需求，题目权限公开实现内容复用 |
| 为什么学习联动简化为看视频得次数？ | 不采用复杂任务完成判定 | 降低实现复杂度，提升用户体验，规则可配置保持灵活性 |

---

## 10. 当前待确认事项

以下事项影响应用架构的细化，需在专项设计前明确：

1. **国家级平台账号打通方式**：单点登录、账号绑定还是静默映射？影响用户中心与协同网关的分工边界。
2. **跨系统同步时效要求**：实时、准实时还是定时同步？影响协同网关的技术实现模式。
3. **统计中心的数据实时性要求**：排名是实时计算还是定时批量？影响统计中心与赛事引擎的交互方式。
4. **题库题目权限公开的粒度**：整题库公开还是单题公开？影响题库引擎的权限模型设计。

## 11. 结论

本文档将灾害信息员大赛平台拆解为**6个能力模块 + 3个用户触达终端**，明确了每个模块的职责边界和协作关系。

本文档的核心约束是：**基于国家培训竞赛平台现有能力进行改造升级**，不是从零新建。因此：

- 题库引擎在现有题库管理基础上扩展分层能力
- 赛事引擎复用现有竞赛逻辑，规则固定不做自定义
- 用户中心在现有用户管理基础上扩展机构树和权限下沉
- 协同网关简化改造，从复杂任务联动改为看视频得次数

后续所有功能设计应遵循以下顺序：

1. 先明确功能属于哪个模块
2. 再明确该功能在哪个终端上呈现
3. 最后展开页面级设计

只有先把模块边界定清，后续跨终端的功能才不会写岔，跨团队的开发才能对齐。
