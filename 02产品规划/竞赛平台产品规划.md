## 功能状态定义

| 状态 | 定义 | 判定标准 | 更新时机 |
|------|------|----------|----------|
| 🔁 可复用参考 | 国家培训平台或现有通用后台已存在的真实能力，可作为本项目重开发的参考来源 | 已有真实页面、后台或链路可参考，但本项目尚未开发完成 | 识别到明确可复用能力时 |
| ❌ 待建设 | 本项目已明确需要开发，但当前尚未完成开发交付 | 需求范围和版本目标已明确，需要基于复用参考做重开发、适配或新增 | 纳入版本建设范围时 |
| ⚠️ 待解决 | 存在影响规划或交付的问题，需要确认或决策 | 业务规则、参数口径、技术方案、外部依赖未明确 | 发现阻塞问题时 |

**状态维护规则**：
- 状态变更时，需同步更新模块内子功能列表和“功能总览”中的汇总数据。
- 标记为`🔁 可复用参考`的子功能，必须写明参考来源和能力边界，避免被误解为本项目已交付。
- 标记为`❌ 待建设`的子功能，需写明建设目标、依赖项和计划版本。
- 标记为`⚠️ 待解决`的问题，需明确影响范围、阻塞功能、责任人和目标时间。

## 业务架构总览

**一句话定位**：
复用国家平台既有竞赛与系统管理底座，为各省提供“练习 + 正式竞赛 + 运营管理 + 国家级观察”的 SaaS 化办赛能力。

### 核心角色

| 角色 | 角色定位 | 核心动作 |
|------|----------|----------|
| 平台运营 | 平台级管理员 | 开通省份、设置有效期、配置赛段/赛制可用范围、维护平台参数 |
| 国家级观察 | 监督与查看方 | 只读查看全国及分省赛事数据，不参与省级配置 |
| 省级运营 | 省级办赛方 | 配置赛事、题库、规则，查看结果、统计和导出 |
| 市级报名单位 | 报名执行方 | 在需要报名的赛事中完成市级报名、队伍组织 |
| 参赛用户 | 练习与竞赛参与者 | 日常练习、参加正式竞赛、查看记录和结果 |
| 国家培训平台 | 外部支撑系统 | 同步用户与学习数据，支撑练习激励发放 |

### 核心业务域

| 业务域 | 业务目标 | 当前建设策略 |
|--------|----------|--------------|
| 省份开通 | 控制哪些省份可用，以及可用到什么时间 | 复用系统管理底座，新增省份开通与有效期能力 |
| 赛事配置 | 配置初赛/复赛/决赛及个人赛/团队赛规则 | 复用现有竞赛配置页面，做省级参数化适配 |
| 练习激励 | 提供练习入口并与培训学习行为联动 | 前台已具备页面壳，后续接通次数和激励规则 |
| 报名参赛 | 控制是否报名、市级报名和本省参赛资格 | 复用现有报名入口与结果链路，补齐校验逻辑 |
| 成绩晋级 | 输出成绩、排名和晋级结果 | 沿用现有能力，晋级结果每天 24 点自动更新 |
| 统计导出 | 支撑省级复盘和国家级观察 | 复用现有统计/导出能力，继续校准口径 |
| 用户与权限 | 承接用户主档、身份、机构和角色 | 培训平台同步数据，竞赛平台做项目化视图适配 |

### 业务协同关系

```text
国家培训平台
  -> 同步用户基础数据、机构归属、灾害信息员等级、课程完成数、学习时长
  -> 支撑竞赛平台中的用户识别和练习激励

平台运营
  -> 开通省份、设置有效期、配置可用赛段/赛制
  -> 省级获得办赛权限

省级运营
  -> 配置赛事、题库、规则和报名方式
  -> 发布后提供给本省用户参与

市级报名单位
  -> 在“需要报名”的赛事中完成市级报名和队伍组织

参赛用户
  -> 先练习，再参与正式竞赛
  -> 产生练习数据、竞赛数据、结果数据

国家级观察
  -> 只读查看全国与分省数据
  -> 不参与省级配置和报名执行
```

## 核心业务流程

### 1. 省份开通与交付流程

```text
平台运营开通省份
-> 设置服务有效期
-> 选择可用赛段/赛制（初赛/复赛/决赛；个人赛/团队赛）
-> 省级后台获得对应配置与运营权限
-> 到期后关闭正式竞赛能力，仅保留练习功能
```

**关键规则**：
- 省份开通只按省级维度控制。
- 练习功能不受省份开通限制。
- 服务到期后，正式竞赛关闭，练习继续可用。

### 2. 练习激励流程

```text
用户进入练习页
-> 消耗基础练习次数
-> 基础次数不足时，查看“获取更多练习次数”
-> 跳转国家培训平台完成课程数/学习时长目标
-> 培训平台同步学习结果
-> 竞赛平台按配置规则发放奖励次数
-> 用户返回练习页继续练习
```

**关键规则**：
- 基础练习次数按配置项维护，按自然日重置。
- 奖励次数发放规则按配置项维护。
- 奖励次数是否叠加、有效期等也通过配置控制。

### 3. 正式竞赛执行流程

```text
省级运营配置赛事
-> 配置初赛/复赛/决赛及个人赛/团队赛规则
-> 配置是否需要报名
-> 如需要报名，则由市级报名
-> 用户按本省资格参与赛事
-> 系统生成成绩、排名和对战结果
-> 每天24点自动更新晋级结果
```

**关键规则**：
- 初赛、复赛、决赛均可配置个人赛和团队赛。
- 同一阶段允许个人赛和团队赛同时开启。
- 团队成员来源于国家平台同步到竞赛平台的用户体系。

### 4. 统计复盘与国家观察流程

```text
赛事执行中/结束后
-> 省级运营查看本省数据统计与导出
-> 国家级观察查看全国汇总与分省数据
-> 省级输出复盘材料
-> 平台沉淀后续赛事配置与运营经验
```

**当前待定项**：
- 省级最小统计指标集是否与国家级监督指标集完全一致。
- 现有导出模板是否足够覆盖省级交付汇报场景。

## 功能实现现状

### 省份开通与权限中心

> **模块定义**：详见 [产品上下文 - 省份开通与权限中心](../01产品上下文/竞赛平台产品上下文.md#省份开通与权限中心)
> **详细功能描述**：待后续按国家平台可复用现状和本项目建设结果逐步沉淀

**模块定位**（引自产品上下文）：
承接省级 SaaS 的省份开通、服务有效期、访问边界和省份隔离控制。

**核心流程**：
```text
平台运营超级管理员设置省份开通状态/有效期
→ 用户访问省份入口
→ 系统校验所属省份与开通状态
→ 放行或拦截
```

#### 模块概览

- 当前阶段：待建设
- 本期重点：补齐省份开通状态、服务有效期、正式竞赛入口拦截
- 前置条件：用户与机构归属信息可用于省份识别
- 关键依赖：系统管理中心、用户与人员中心
- 当前负责人：产品负责人 / 技术负责人

#### 可复用参考能力

> 本模块当前暂无可直接参考的完整省份开通能力，仅存在可借用的通用权限底座。

#### 待建设子功能

**权限底座承接** ❌ 待建设
- 功能说明：复用现有机构管理、角色管理、账号管理作为省份开通与访问控制底座
- 可复用参考：国家平台中的机构树、角色、账号等后台基础能力已存在，可用于后续承接省份隔离与访问控制
- 建设内容：在复用通用后台基础上重新开发“省份开通状态 + 服务有效期 + 前台拦截规则”的完整闭环
- 能力边界：当前仅能支撑底座复用，不能直接等同于省级 SaaS 的开通管理能力
- 优先级：高
- 目标版本/时间：V1.0 / 2026Q2
- 前置条件：机构层级、用户归属字段可稳定识别省份
- 关键依赖：系统管理中心、用户与人员中心
- 当前负责人：技术负责人

#### 后续待建设子功能

**省份开通状态管理** ❌ 待建设
- 功能说明：支持平台级按省份配置开通/关闭状态
- 业务价值/触发原因：支撑“一省一赛”交付和签约开通管理
- 预期核心能力：按省份查询、启停、查看当前服务状态，并可联动控制初赛/复赛/决赛中个人赛或团体赛的开启范围
- 能力边界：仅按省级维度管理 SaaS 服务状态和可用赛段/赛制范围，不替代赛事配置，也不单独区分国家级观察场景的开通口径
- 优先级：高
- 计划版本/时间：V1.0 / 2026Q2
- 前置条件：现有机构树可稳定识别省份维度
- 关键依赖：系统管理中心
- 当前负责人：产品负责人

**服务有效期管理** ❌ 待建设
- 功能说明：为每个已开通省份配置服务有效期
- 业务价值/触发原因：支撑合同期控制、到期提醒和到期停服策略
- 预期核心能力：有效期设置、到期预警、到期后仅保留练习功能可用
- 能力边界：仅控制平台服务期，不处理商务续费流程
- 优先级：高
- 计划版本/时间：V1.0 / 2026Q2
- 前置条件：商务开通流程口径已确认
- 关键依赖：系统管理中心
- 当前负责人：产品负责人

**正式竞赛入口拦截** ❌ 待建设

### 技术约束与规划影响

- 当前可复用参考：现有国家平台竞赛后台、通用系统管理后台、已完成的小程序页面方案
- 当前主要约束：技术栈和部署细节当前材料未披露，正式规则里仍有若干关键参数待定
- 新功能设计注意事项：
  - 默认优先复用国家平台现有页面和后台能力
  - 练习能力与正式竞赛权限边界要区分处理
  - 所有省级 SaaS 适配都需要考虑省份隔离、角色隔离和审计留痕
  - 涉及培训平台联动的能力必须参数化设计，避免把暂定业务规则写死
